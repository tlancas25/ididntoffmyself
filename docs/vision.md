# Vision — why REDFORGE exists

> © 2026 BlaFrost Softwares Corp. Internal.

## The thesis

Every security tool on the market today is a variation of the same idea: take a known set of vulnerability patterns, match them against input, emit alerts. It is 2026 and this approach has not fundamentally changed in twenty years.

Meanwhile:

- Software is shipping faster than at any point in history, and an increasing percentage of it is **vibe-coded** — generated by LLMs, assembled quickly, and deployed before any rigorous review.
- Attack surfaces are multiplying. Every app now talks to LLMs. Every LLM now calls tools. Every tool now has privileges. The confused-deputy chains between these components are a zero-day factory.
- **Attackers already use AI.** Defensive tooling that doesn't use AI is fighting a machine gun with a bow.
- Posture management is no longer a feature in a SaaS dashboard. It is the *product* — every organization's critical assets need continuous adversarial evaluation.

REDFORGE is our bet that the right answer is **to be the AI attacker first** — before the adversaries are. Instead of a static scanner, we field a **team of specialist agents** that read, reason, hypothesize, test, and chain findings. The platform operates with the methodology of a senior pentesting team and the speed of software.

## Why "preventative cyber" and not "reactive"

Traditional security spending is concentrated in reactive posture: SIEM, SOAR, EDR, IR retainers, breach lawyers. Every dollar spent reactively is a dollar spent *after* an incident has already cost the organization something real.

Preventative cyber is structurally underinvested because it's hard to do well. It requires:

1. **Specialist depth across many surfaces** (auth, authz, injection, SSRF, business logic, LLM abuse, agent autonomy, supply chain, infra, credentials, persistence, network posture — ~20 distinct domains).
2. **A way to chain findings across specialists** — the highest-impact bugs are chains, not single primitives.
3. **Continuous iteration** — the target surface evolves every sprint.

A single human pentester can do #1 shallowly or one domain deeply. A firm can do #1 and #2 for $50k–$250k per engagement, once a quarter. Neither scales.

REDFORGE makes (1) and (2) machine-native and (3) continuous. That's the market.

## What "catching zero-day exploits" means to us

We are explicit internally: we will not catch every novel CVE. Claiming that would be dishonest and customers would figure it out fast. What we CAN do, and what we build the platform to do:

1. **Reduce the class of zero-day-adjacent bugs that ship.** Most "zero-days" are chains of known primitives that no single tool cross-referenced. We explicitly model chains.
2. **Catch LLM-era emergent patterns** that traditional tools are blind to: prompt injection, indirect injection via RAG, LLM-output-trust (confused deputy via model-generated strings), MCP tool-permission abuse, agent-autonomy goal-drift.
3. **Build a living detector corpus.** Every confirmed novel pattern from a real trial becomes a new detector for the next trial. The platform's capability compounds. This is the moat.
4. **Surface hardening gaps as a prioritized action list.** Every finding carries a concrete `hardening_plan`: immediate one-liner, durable config, monitoring rule, compensating controls, estimated effort. The report is a work queue, not a PDF.

## The product family

REDFORGE is a platform. Specific products we build on top:

1. **`rfatk` CLI (this repo, `rfatk-cli/`)** — the flagship agentic red-teamer. Distributable to customers with BYO-API-key. Supports 12 LLM providers. Code-scan and host-scan modes.

2. **`redforge-claude-code` (this repo, `claude-code-native/`)** — the Claude-Code-native distribution BlaFrost engineers use internally. Uses operator's Claude Max seat. Same methodology, different delivery.

3. **Detector library** (future, separate repo) — the curated rules distilled from trial findings. Exports as SAST rules, DAST payloads, Suricata signatures, Sigma rules, Sysmon hunts.

4. **Continuous posture product** (future, SaaS) — scheduled re-scans, diff reports, regression alerts. Sold as a managed offering.

5. **Incident-response hunt mode** (future) — same specialist roster, repointed at "is this machine already compromised?" Uses the host-scan sandbox; emphasizes persistence-hunt + credentials-exposure + alert-triage.

Each product reuses the core prompts, schema, and orchestration. The separation between `claude-code-native/` and `rfatk-cli/` in this repo is the foundation for all five.

## Ownership and posture

BlaFrost Softwares Corp owns everything in this repo, period. Not open-source. Not for public use. No licensing, no redistribution, no derivatives.

The choice of proprietary licensing is strategic: the platform's value is its methodology and its iterative detector corpus. Open-sourcing the prompts would let competitors catch up overnight. Open-sourcing the CLI would confuse the commercial product story. We will open-source specific components *later*, tactically, when it serves the business (e.g. SDK examples, a community-edition specialist for PR purposes).

For now: keep it locked. Every commit is internal.
